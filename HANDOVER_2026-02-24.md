### Key Findings

1. **Synthesis LLM call is 70-85% of total latency.** Steps 1-4 (embed, triple/DPR retrieval, seed building, PPR) take ~1 second total. Step 5 (gpt-4.1 synthesis) takes 4-12 seconds. Latency is almost entirely proportional to context token count.

2. **Context tokens nearly doubled after chunk merging** (commit `98e717c`): from ~7,200 tokens (18 unmerged chunks) to ~14,941 tokens (13 merged chunks). Merged chunks produce mega-blocks because the entity-matching ±1 sentence windows cover nearly the entire merged text when entities are frequent.

3. **The `ROUTE7_MAX_CHUNK_CHARS=7000` cap trims 2 quality points** by truncating the Builders Limited Warranty chunk_0 (15,068 chars → ~7,000 chars), which removes arbitration subsections 9B-9G containing confidentiality, small claims, venue, and cost-splitting provisions needed by Q-G5.

## 2. Root Cause Analysis: The 2-Point Regression (55→53)

### Q-G5: "What remedies / dispute-resolution mechanisms are described?" (3/3 → 2/3)

- **Cause: Retrieval truncation.** The Builders Limited Warranty chunk_0 is 15,068 chars. With `ROUTE7_MAX_CHUNK_CHARS=7000`, the latter half is cut off, removing arbitration subsections 9B-9G (confidentiality, self-help remedies, provisional remedies, $3,000 small claims threshold, Pocatello venue, 180-day timeline).
- **The entity-matching window covers the early sections** (Term, Exclusions) because entities like "Builder", "Idaho", "Owner" appear there. The later subsections 9B-9G get trimmed by the character cap.
- **Q-G5 has the smallest context of all questions** at 7,374 tokens (all others are 10,000-11,400), confirming aggressive truncation.

### Q-G6: "List all named parties/organizations" (3/3 → 2/3)

- **Cause: LLM generation variability.** Both runs had the same 13 chunks and similar context. The cap run's LLM incorrectly attributed "Authorized Representative" as a party role (it's a signature block title), which the judge penalized. The entity-document map fed "Authorized Representative" as an entity seed, and the LLM over-indexed on it.
- **This is stochastic** — Q-G6 was historically unstable (1/3 on Feb 22-23, 3/3 starting Feb 24 12:55).

### Q-G4: "What obligations does each party have under Document 2?" (persistent 1/3)

- **Not related to caps.** Q-G4 scores 1/3 across ALL configurations. The judge penalizes over-inclusion of obligations. This is a prompt/synthesis issue, not retrieval.

## 3. Root Cause Analysis: The Latency Regression (5s → 10s)

| Factor | Pre-merge (Feb 22) | Post-merge (Feb 24) |
|--------|---------------------|----------------------|
| Chunks sent to LLM | 18 small | 13 merged |
| Avg context tokens | ~7,200 | ~14,941 |
| Avg chunk size | ~400 tokens | ~1,150 tokens |
| p50 latency | 3,938ms | 8,215ms |
| Synthesis model | gpt-4.1 | gpt-4.1 |

**The merging step (commit `98e717c`) is the sole cause.** It merges adjacent same-section chunks, combining their sentence arrays. When `_focused_text` runs entity matching over the merged (2x longer) sentence array, entities match across many sentences, and the ±1 expansion windows merge into one huge window covering most of the merged text. Result: each merged chunk contributes ~1,150 tokens instead of ~400 tokens.

## 4. What Has NOT Been Tried Yet

### A. Smart truncation instead of hard character cap

Instead of blindly cutting at 7,000 chars, truncate by **limiting the number of entity-match windows** per chunk. The ±1 windows expand and merge — capping to e.g. the top-3 highest-scoring windows (by entity frequency) would preserve the most relevant content while reducing size.

### B. Increasing PPR_PASSAGE_TOP_K with no merging

Revert chunk merging, keep 18 separate chunks, and increase `ROUTE7_PPR_PASSAGE_TOP_K` from 20 to e.g. 25 to compensate for any section-coherence benefit that merging provided.

### C. Adjusting the HippoRAG2 algorithm parameters

These have NOT been tuned at all — all are at defaults:

| Parameter | Current | What it does | Tuning opportunity |
|-----------|---------|-------------|-------------------|
| `ROUTE7_DAMPING` | 0.5 | PPR damping factor | Lower (0.3-0.4) = more exploitation of seed nodes, possibly more focused retrieval |
| `ROUTE7_PASSAGE_NODE_WEIGHT` | 0.05 | Weight of entity↔passage edges | Higher = passages rank higher in PPR, possibly fetching more relevant ones |
| `ROUTE7_SYNONYM_THRESHOLD` | 0.8 | Min similarity for synonym edges | Lower = more entity connections, broader reach |
| `ROUTE7_TRIPLE_TOP_K` | 5 | Triples before LLM filter | More triples = better entity seeding but slower |
| `ROUTE7_DPR_TOP_K` | 20 | DPR passage results | Could reduce to limit chunks and context |
| `ROUTE7_DPR_SENTENCE_TOP_K` | 60 | Raw sentence candidates | Lower = faster DPR but possibly fewer chunks |
| `ROUTE7_PPR_PASSAGE_TOP_K` | 20 | Passages from PPR output | Lower = fewer chunks = less context = faster |
| `_MAX_MERGE` | 2 (hardcoded) | Max merge chain length | 1 = disable merging entirely |
| `_MAX_FALLBACK_SENTS` | 8 (hardcoded) | Cap for non-entity chunks | Confirmed to hurt quality at 8; currently deployed as 0 via env var |

### D. Reducing PPR_PASSAGE_TOP_K specifically

Currently 20 passages are fetched, which after merging become 13. Reducing to e.g. 15 would produce ~10 merged chunks, reducing context substantially. This is the most direct lever for latency and has not been tested.

### E. Focused-text window size

The current ±1 expansion (3-sentence window) is hardcoded. Reducing to ±0 (exact sentence only) would dramatically reduce chunk sizes but may lose context. Worth testing.

### F. Purchase contract chunk deduplication

Citation [10] in the cap=7000 run contains the purchase contract text repeated 3 times within one chunk, wasting ~2/3 of that chunk's token budget on duplicate content. This is a data/indexing issue, not a retrieval issue.

## 9. Architecture Quick Reference

```
Query → Step 1: Voyage embed (220ms)
      → Step 2: Parallel { Triple match + DPR sentence search } (735ms)
      → Step 3: Build entity seeds from triples + DPR (0ms)
      → Step 4: PPR over knowledge graph (4ms)
      → Step 5: Synthesis via gpt-4.1 (4,000-12,000ms)  ← BOTTLENECK
```

The PPR graph contains: Entity nodes, TextChunk (passage) nodes, edges (RELATED_TO, MENTIONS, SEMANTICALLY_SIMILAR). PPR runs power iteration with damping=0.5, returns top-K passage nodes ranked by PPR score. Passages are fetched from Neo4j, merged by section, focused by entity-matching ±1 sentence windows, and sent as context to gpt-4.1 for synthesis.
